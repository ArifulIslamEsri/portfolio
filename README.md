## Why This Portfolio Matters to a PE Team

Most junior engineers haven't shipped real products to real users. I have something different.

**I've watched hundreds of developers struggle with the same SDK** — and I know exactly what mental model they're missing. My projects aren't ideas from a tutorial. They're the answers I wish the docs had when developers were stuck at 2am.

Specific things a Maps SDK PE team would care about:

- **GeoSight** was born from noticing ~40% of JS SDK tickets were about syncing `FeatureLayer` edits with real-time data. The app is my proposed DX pattern — clean, under 300 lines.
- **DevEdge** was born from the #1 developer complaint I hear: *"the error message doesn't tell me what to do."* Every fix suggestion maps to a real ticket category.
- **LayerLens** was born from "why is my layer not showing?" — asked across all three SDK surfaces daily.
- **RouteBoard** shows end-to-end product thinking: map interaction → REST API → results back on map. Non-technical users can operate it without seeing a line of SDK code.
